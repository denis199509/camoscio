ORIGINE DEL PROGETTO
---------------------
"Camoscio" è un'app social per l'escursionismo in montagna (Marche, Lazio,
Abruzzo, Molise). L'utente (proprietario del progetto) ha scritto le sue
richieste in `cose_da_fare.txt` (nella stessa cartella di questo file) —
quel file resta la fonte originale dei requisiti, non cancellarlo.

Riassunto delle richieste originali (cose_da_fare.txt):
1. Registrazione utenti reale con molti campi (vedi dettaglio Fase C sotto)
2. Repository GitHub + pubblicazione online + migrazione a MongoDB
3. Tracciamento GPS live durante l'escursione, solo dati utili, efficiente
   con poco campo in montagna, mappa scaricata offline PRIMA di partire
4. Il GPS deve restare "agganciato" al sentiero (tolleranza ~10m errore OSM),
   se non esiste un sentiero noto va mappato per il futuro
5. Nuova palette colori "montagna" (verde, marrone chiaro)
6. Caccia ai bug generale
Vincoli: MongoDB con formato dati efficiente (poco spazio), tutto open
source, mantenere i 4 account demo esistenti accessibili da una pagina
dedicata (Marco Alpinista, Luca TrailRunner, Sofia FotoPasso, Giulia
Escursionista), regione geografica limitata a Marche/Lazio/Abruzzo/Molise.

- Nome file per questo promemoria: leggimi.txt (richiesto esplicitamente
  dall'utente, in italiano, dentro la cartella del progetto)
- Libreria "osmtogeojson" (prevista in origine per la Fase G) NON installata:
  ogni versione pubblicata su npm porta con sé una libreria di lettura XML
  (xmldom) con vulnerabilità di sicurezza critiche irrisolte (gravità
  massima, 9.8/10, su tutte le versioni disponibili). Per la Fase G si userà
  invece l'output nativo in formato JSON di Overpass API (si può richiedere
  direttamente, senza passare da XML) con una piccola funzione scritta a
  mano per trasformarlo in GeoJSON — stesso risultato, zero dipendenze
  rischiose in più.
- Stringa di connessione MongoDB: usa il formato "standard" (mongodb://...
  con gli host dei 3 shard elencati uno per uno), NON il formato
  "mongodb+srv://" che Atlas mostra di default. Motivo: su questo PC
  qualcosa (quasi certamente AVG o McAfee, entrambi installati insieme
  a Windows Defender) blocca specificamente le query DNS di tipo SRV
  fatte da Node.js, mentre le query DNS normali e le connessioni TCP
  dirette funzionano bene. La stringa attuale è dentro .env, con gli
  host espliciti (ac-fh8vbm9-shard-00-00/01/02.shlr4hq.mongodb.net:27017)
  e il nome del replica set (atlas-aley8a-shard-0) letti a mano dal DNS.
  RISCHIO DA SAPERE: se in futuro Atlas cambia gli host del cluster
  (succede raramente, es. durante manutenzioni), questa stringa andrà
  rigenerata a mano (rifare la stessa procedura: Resolve-DnsName -Type
  SRV/TXT sul nome del cluster, oppure capire perché Node non riesce a
  fare query SRV su questo PC e risolvere quello — es. aggiungendo
  un'eccezione per node.exe nel firewall di AVG/McAfee).
- Identità autore Git per i commit: nome "Denis", email
  denis1995.09@gmail.com — impostata SOLO su questo repository (git
  config locale, non globale), quindi non tocca altre impostazioni git
  eventualmente presenti sul PC per altri progetti.

STATO DEL CODICE PRIMA DI QUESTO LAVORO (come si presentava il progetto)
--------------------------------------------------------------------------
- Stack: Node.js + Express 4 + libreria "ws" per WebSocket. Frontend: un
  solo public/index.html (SPA con sezioni mostra/nascondi), 8 file JS in
  public/js/ (app.js, map.js, profile.js, safety.js, social.js, backpack.js,
  carpool.js, weather.js), nessun framework, nessun bundler.
- Dati: un solo file data/db.json, letto/riscritto per intero ad ogni
  richiesta tramite readDb()/writeDb() dentro server.js (righe 220-241).
  NESSUN MongoDB.
- Autenticazione: NESSUNA VERA. Esisteva POST /api/login ma era codice
  morto (nessun frontend lo chiamava). L'"utente corrente" era solo una
  scelta da un menu a tendina nell'header, salvata in localStorage
  (chiave "camoscio_user_id"), default "user_marco". Zero verifica lato
  server — qualunque endpoint si fidava dell'ID utente mandato dal client.
- I 4 account demo (Marco/Luca/Sofia/Giulia) erano semplicemente le uniche
  4 righe nella tabella utenti di data/db.json.
- GPS/tracciamento: NESSUNO reale. Il percorso disegnato durante
  un'escursione era 5 punti finti generati a tavolino attorno al
  trailhead (solo per la demo dell'esposizione solare). Un pulsante
  "usa GPS reale" chiamava getCurrentPosition() UNA SOLA VOLTA (non
  watchPosition) e non mandava nulla al server.
- Aggancio al sentiero: NESSUNO. Nessuna libreria di geometria oltre a
  Leaflet stesso (niente Turf.js).
- Mappa offline: NESSUNA cache. Le tile OpenStreetMap si scaricavano
  sempre dal vivo, nessun service worker.
- Vincolo regionale: window.CAMOSCIO_REGION_BOUNDS in public/js/map.js,
  un semplice rettangolo {minLat:40.8, maxLat:43.9, minLng:11.4,
  maxLng:15.2}, applicato SOLO lato client (maxBounds di Leaflet +
  validazione nel form crea-escursione). Zero controllo lato server.
- Colori: centralizzati in :root dentro public/css/styles.css (righe
  4-32, tema "Dark Alpine": sfondo blu navy scurissimo #0B0F19, accento
  arancione neon #FF6B35) ma ANCHE duplicati a mano dentro map.js e
  app.js in vari punti (marker, popup, grafici Chart.js).
- Bug noto trovato durante l'esplorazione (da correggere in Fase H):
  resolveReportDirectly() in map.js "risolve" una segnalazione POSTando
  di nuovo su /api/reports, ma quella rotta fa solo un push di un nuovo
  documento → crea un DUPLICATO invece di aggiornare l'originale, che
  resta sempre "attivo".
- Ambiente PC (verificato dal vivo prima di iniziare): Node.js v24.18.0
  e npm già installati e funzionanti. Git NON installato. Python NON
  installato (solo stub Windows Store). Nessun MongoDB locale. La
  cartella non era ancora un repository git.
